# GitHub is authoritative for the local Issue projection

GitHub is the source of truth for Ready-labeled Issues and their hierarchy; the local Issue store is a rebuildable projection containing only the Relevant Issues that form the harness working set. The Issue Reconciler is its sole application-level writer and reconciles one Repository at a time from a complete GitHub traversal, deriving relevance before upserting new or changed Issues and hard-deleting Issues that are no longer Relevant. This deliberately omits closed roots, children of closed or non-Ready-labeled parents, and every member of unsupported cross-Repository or deeper-than-one-level hierarchies; a closed direct child of an open Ready-labeled parent remains in the working set. Draft pull requests do not affect relevance; an open non-draft or merged closing pull request makes an Issue irrelevant unless it belongs to a harness Work Item. We chose this one-way filtered projection over bidirectional synchronization, tombstones, mirroring every GitHub Issue, and retaining irrelevant Ready-labeled Issues because control remains in GitHub and downstream consumers need one consistently pruned working set.

Database mutations are deliberately sequential and non-transactional: upserts run before deletions, partial progress is reported as a typed failure, and an external caller retries until the projection converges. A failed or unavailable GitHub request makes no database changes, while drift caused by changes during pagination is accepted until a later reconciliation. Each Repository records when reconciliation last completed successfully; failed or partial runs do not advance that time.
