# GitHub is authoritative for the local Issue projection

GitHub is the source of truth for Ready-labeled Issues; the local Issue store is a rebuildable projection used to avoid repeated GitHub API calls. The Issue Reconciler is its sole application-level writer and reconciles one Repository at a time from a complete GitHub traversal, retaining closed Issues that still carry the label, upserting new or changed Issues, and hard-deleting Issues absent from the fetched set, including Issues whose label was removed. We chose this one-way model over bidirectional synchronization, tombstones, and mirroring every GitHub Issue because control remains in GitHub and the harness only needs the Ready-labeled working set.

Database mutations are deliberately sequential and non-transactional: upserts run before deletions, partial progress is reported as a typed failure, and an external caller retries until the projection converges. A failed or unavailable GitHub request makes no database changes, while drift caused by changes during pagination is accepted until a later reconciliation. Each Repository records when reconciliation last completed successfully; failed or partial runs do not advance that time.
