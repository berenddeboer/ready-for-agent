# Persist Work Items as a transactional lifecycle

An explicit Implement Now request creates a durable Work Item, separate from the rebuildable Issue projection, with at most one unfinished Work Item per Issue and a Step Run for every scheduled attempt. A dedicated Work Item Lifecycle module owns compare-and-start execution, manual retries, abandonment, timeout and interruption outcomes, typed step context, and atomic completion, advancement, and next-step enqueue; lifecycle jobs therefore require the database-backed transactional queue rather than an external queue without an outbox. This design preserves timings and failure history under at-least-once delivery while acknowledging that injected step handlers must be re-entrant because database transactions cannot make external effects exactly once.
