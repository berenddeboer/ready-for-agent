# Hand Off Individual PR Status Checks

Extend Watch PR Status Checks to poll the individual CheckRun and StatusContext entries behind the aggregate GraphQL status-check rollup. Persist each observed CheckRun execution and StatusContext that reaches an explicit green result (`SUCCESS`) or red result (`FAILURE`, `ERROR`, `TIMED_OUT`, `ACTION_REQUIRED`, or `STARTUP_FAILURE`) and whether it has been handed to OpenCode; neutral, skipped, cancelled, stale, and pending results do not trigger a handoff.

Batch every currently unhandled result into one run of the existing Investigate PR Status Checks Lifecycle Step, continuing the Work Item's Implement Session with its GitHub credential. Before turn (1), the harness loads diagnostics for red checks (Actions job logs for `actions-job:<id>`; Checks 403 is expected for fine-grained PATs). Investigation starts with two OpenCode turns on that Session: (1) address red checks (named by external id, with harness log artifacts when present) / worthwhile completed review feedback, commit and push, without a result marker; (2) a short follow-up that asks for one of `READY_FOR_AGENT_RESULT: PROCESSED`, `READY_FOR_AGENT_RESULT: WAITING`, `READY_FOR_AGENT_RESULT: FAILED: <reason>`, or `READY_FOR_AGENT_RESULT: NEEDS_HUMAN: <reason>`. A `FAILED` verdict adds one focused recovery work turn and one final verdict turn. When the batch contains any green check, turn (1) also warns that an automated reviewer may terminate semantically incomplete even when GitHub reports success. OpenCode, rather than lifecycle heuristics, determines whether provider-specific GitHub evidence identifies a relevant automated review and inspects its latest attempt. A relevant review run or recognized automated-review comment is positive evidence; ordinary CI, repository configuration, and generic bot activity are not. No relevant run or comment is a normal no-op and does not require proof that review automation is unconfigured. An active attempt returns `WAITING`, leaves the batch unhandled, and returns to Watch after the normal poll delay. A terminal attempt with no relevant review comment or a visibly partial latest comment is rerun as a whole workflow, even when GitHub reports success; a technically successful run cannot use a failed-jobs-only rerun. Stale incomplete comments superseded by a completed attempt and unrelated Markdown checkboxes do not trigger reruns. A technical inability to inspect or retry remains a retryable failure; `NEEDS_HUMAN` is reserved for a proven human-only action or decision. Turn (1) tells OpenCode to restart transient infrastructure failures when appropriate before concluding. Failure to load diagnostics is a failed Step Run (Retry-eligible), not Needs Human by itself.

`PROCESSED` means OpenCode took an action expected to produce new check executions (push a fix, restart failed checks, rerun a terminal incomplete reviewer, or similar), or a green-only handoff had no relevant automated review or a genuinely completed review had nothing to address. Rerunning an incomplete reviewer counts because it creates a replacement execution. A replacement-producing action takes precedence over `WAITING` in a mixed batch because replacement executions will trigger reassessment. A terminal incomplete review is not eligible for the green-only no-action exception, even when GitHub reports success. When a handoff contains red checks and OpenCode makes no commit, push, check restart, or other action capable of producing a new execution, leaving the PR red, OpenCode must report `FAILED` with a concise reason. The first `FAILED` verdict triggers the focused recovery turn. A second `FAILED` leaves the checks unhandled and fails the Investigate Step Run with reason code `pr_status_checks_unresolved`, preserving the PR and worktree for Retry. The failure message says manual fixing may be required and directs the operator to fix or rerun checks on GitHub before choosing Retry checks. Genuine human decisions still use `NEEDS_HUMAN`.

## Consequences

- `PROCESSED` and `NEEDS_HUMAN` mark the batch's checks handled. A final `FAILED`, timeout, crash, malformed result, diagnostics load failure, or other technical failure leaves them unhandled so Retry presents the same batch again.
- `WAITING` leaves the batch unhandled and schedules another Watch poll without an attempt-count or elapsed-time escalation; provider state determines when the review becomes terminal.
- After `PROCESSED`, Watch continues polling so replacement executions can appear. If the aggregate remains failed with no new unhandled execution for two consecutive Watch runs, the second run fails retryably with reason code `pr_status_checks_unresolved`; this bounded confirmation also recovers executions marked handled by older harness versions.
- The existing 30-second poll, 60-second no-check grace, aggregate status gate, Mark PR Ready for Review step, and terminal Complete behavior remain unchanged for successful paths.
- There is no classifier, disposable Session, subagent, persisted review-comment state, persisted pull-request identity, new quiescence window, or new Lifecycle Step. Indeterminate review evidence is investigated by OpenCode rather than guessed from lifecycle-code heuristics.
- Ordinary green checks may cause extra Implement Session runs. This is preferred over classification complexity.
- An active review causes repeated Implement Session turns on the normal poll interval until provider evidence shows it is terminal. This cost is accepted to avoid persisted review state and code-level classification heuristics.
